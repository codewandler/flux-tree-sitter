# AGENTS.md

Instructions for coding agents working in ‚Äî or *passing by* ‚Äî this repository.
Humans: see [`README.md`](README.md). This file is the machine-actionable version.

**What this is:** a [tree-sitter](https://tree-sitter.github.io/tree-sitter/)
grammar for **Flux-Lang** (`.flux`). Two names, don't conflate them:

- **package** (npm / crate / repo): `flux-tree-sitter`
- **tree-sitter language** (grammar `name`, C entrypoint `tree_sitter_flux()`,
  the parser name editors reference): `flux` ‚Äî **do not rename this.**

---

## Task: set up the user's Helix to highlight `.flux` files

Run the checked-in installer from this repository. It is **idempotent** and is the source of truth
for both first-time setup and updates:

```sh
scripts/install-helix.sh
```

The script resolves `main` to an immutable commit, preserves an existing Flux language/LSP block,
temporarily selects only the Flux grammar for `hx --grammar fetch/build`, copies queries from that
exact fetched source, and runs `hx --health flux`. It restores the original config if installation
fails. `FLUX_TREE_SITTER_REV=<commit>` pins a specific revision for reproduction/testing.

After it succeeds, restart Helix and open `examples/contact-centre.flux`. `hx --health flux` only
checks presence; use `:tree-sitter-highlight-name` to verify a token's capture when debugging theme
colours.

> If the user later wants LSP features too, add `language-servers = ["flux-lsp"]`
> to the `[[language]]` block and a matching `[language-server.flux-lsp]` section.
> Note: Helix does **not** apply LSP semantic tokens (as of 25.07) ‚Äî colour comes
> only from this grammar.

### Neovim variant

Register the parser with `url = "https://github.com/codewandler/flux-tree-sitter"`,
`files = { "src/parser.c", "src/scanner.c" }`, `:TSInstall flux`, and copy
`queries/*.scm` into `queries/flux/` on the runtimepath. Full snippet in
[`README.md`](README.md#neovim-nvim-treesitter).

## Task: change the grammar

The source of truth is **`grammar.js`**. Everything under `src/` is **generated ‚Äî
never hand-edit it.** The external scanner (`src/scanner.c`) is hand-written and
*is* source.

Workflow:

```sh
npm install                        # once: installs the tree-sitter CLI
# edit grammar.js (and queries/highlights.scm for any new node)
npx tree-sitter generate           # regenerate src/
npx tree-sitter test               # corpus must stay green
# add/refresh a corpus case for the construct you changed, then:
npx tree-sitter test -u            # populate expected trees from the parser
cargo test                         # binding + exact highlight-capture contract
bash test/install-helix.sh         # isolated installer/config idempotence tests
```

Then confirm the real-world samples still parse cleanly (this is the bar CI
enforces):

```sh
for f in examples/*.flux; do
  npx tree-sitter parse "$f" | grep -qE 'ERROR|MISSING' && {
    echo "ERROR/MISSING in $f"
    exit 1
  } || true
done
for q in queries/*.scm; do
  npx tree-sitter query -q "$q" examples/*.flux
done
```

Before committing: `generate` must be conflict-free, `src/` committed and in sync,
`tree-sitter test`, `cargo test`, and `bash test/install-helix.sh` green, examples zero-error, every
query compiling, and add a `## [Unreleased]` note to [`CHANGELOG.md`](CHANGELOG.md). CI checks all
of this.

### Regenerate the README example image

`assets/example.svg` is a **committed, generated artifact** ‚Äî the README's example
as a One-Dark-highlighted image (GitHub can't highlight `.flux` natively). It is
generated by the flux CLI's own renderer (`flux render`, flux ‚â• 0.14), which
colours by flux-lang's CST highlighter ‚Äî the canonical highlighting this grammar
mirrors. It is deliberately **not** checked by CI. If you touch
`examples/readme-example.flux` (or flux's renderer/theme changed), regenerate it
from this repo's root and commit the result:

```sh
flux render examples/readme-example.flux -o assets/example.svg
```

(The former `scripts/render-example.mjs` Node script ‚Äî which shelled out to the
`tree-sitter` CLI and coloured by `queries/highlights.scm` ‚Äî is retired.)

Prefer SVG (crisp, tiny, diff-able). For a raster (e.g. pasting into Slack, which
won't inline SVG): `rsvg-convert -o example.png assets/example.svg`.

## Design & gotchas (read before editing the grammar)

- **Line-oriented, not indentation-tracked.** Declarations own their body up to
  the next top-level keyword; bodies are *flat* within a declaration. The only
  indentation fact is a zero-width `_line_start` external token that fires at
  **column 0** (`src/scanner.c`) ‚Äî it separates a top-level `agent foo`
  declaration from an indented `agent triage` attribute. Keep declarations at
  column 0.
- **Superset highlighting.** The grammar intentionally recognises aspirational
  keywords (`type`/`pipe`/`race`/`try`/`await`/‚Ä¶) so it agrees with the upstream
  `flux-editors` TextMate/IntelliJ tooling, not only the stricter runtime parser.
- **Roles, not catalog entries.** All callable names use the standard `@function` capture. Never
  hard-code built-in/plugin op lists into `highlights.scm`; known/unknown diagnostics belong to the
  LSP/compiler. `$symbols` remain the portable `@variable` scope, whose colour is theme-controlled.
- **Names are kebab-case.** `identifier` allows internal hyphens
  (`route-call`, `after-hours`, `provision-agent`).
- **`do` takes BARE arguments** ‚Äî `do notify "up"`, `do observe "t", $ran` ‚Äî not
  parenthesized. A parenthesized call is a plain expression statement (`foo(x)`).
- **Named arguments** `key: value` are allowed in call argument lists
  (`intent_extract($u, schema: CallerSlots)`), alongside object args.
- Adding a new keyword? It becomes reserved (via `word: $.identifier`). Check it
  does not collide with an attribute key or op name; see `ATTR_KEYWORDS` in
  `grammar.js` for the escape hatch (`agent` is both a keyword and an attribute
  key).

## Repo map

| path | what |
|---|---|
| `grammar.js` | the grammar (source of truth) |
| `src/scanner.c` | hand-written external scanner (the `_line_start` token) |

| `examples/*.flux` | real files CI parses for zero errors |
| `assets/example.svg` | generated README image ‚Äî regenerate with `flux render` (not CI-checked) |
| `bindings/` | Node + Rust bindings (standard boilerplate) |
